Introduction:

Song Similarity is a popular Machine Learning application that uses methods like collaborative filtering, frequent pattern mining, and embedding inference (​​Bonnin et al., 2014).

Methods can be audio-based, like  “Song2vec”  (Ross et al., 2017) wherein a generative adversarial network was used to map data-heavy audio clips to a 64 length vector. Using this embedding, they were able to cluster similar songs better than embeddings that reduce dimensionality using PCA. Conversely, methods can be attribute based, like “Steerable Playlist Generation” (Maillet et al., 2009), which generates recommendations from a seed song using a vector with song characteristics like acousticness, tempo, etc. The model compares potentially similar songs to an input tag, a vector of music-relevant words, and minimizes the distance between the candidate songs and the tag to finalize the song that most matches the user’s described context.

Problem + Solution:

Most recommendation systems fail to allow users to influence the direction of the playlist, instead using listener history to generate song recommendations. Our song recommendation model enables users to specify a social tag along with their seed songs to influence the completion of their playlist. The social tag will allow our model to account for characteristics that cannot be conveyed through audio features and allow users to generate a steerable playlist.

Dataset

We have predetermined four distinct social tags (party, chill, sad, and happy) that the user can input in order to generate their steerable playlist.

First, we need to create four separate datasets, one for each social tag, to train our model on. To create a training dataset for each tag, we called the Spotipy Web API on a compilation of Spotify playlists that represent the social tag. For example, we combined songs from several ‘Spotify Chill Mix’ playlists for the chill tag dataset.

For each song, Spotipy provides vector representations (seen below) of:

  • song characteristics: danceability, energy, key, loudness, mode, speechiness, instrumentalness, liveness, valence, tempo, acousticness, duration, and time_signature
  • song identifiers: artist, album

Figure 1: The dataset obtained from the Spotipy API, showing the features of each track.

The processed dataset for each tag is a dataframe of 1000 songs (randomly sampled from playlists representing that tag) and its 13 normalized features (song characteristics). Here, we are left with four datasets of size (1000, 13) .

Visualizing the Data

First, we ran PCA on our dataset and plotted the first three principal components to see if songs of each tag were sufficiently separable from the other tags.

Data Processing: Feature Selection

While 13 features are helpful for song similarity analysis, we found that it can be computationally expensive and redundant because not all features are important in distinguishing a song.

Upon first glance at the features, we decided not to include ‘key’, ‘mode’, and ‘time signature’, as they are discrete values. We also removed ‘liveliness’, which describes the presence of an audience during a song, and is irrelevant to our project since we do not have any “live” songs in our training datasets. We also removed ‘duration’ because it isn’t a song characteristic that will impact the social tag. Now each tag dataset is of size (1000,8).

Data Processing: Dimensionality Reduction

For further data pre-processing, our goal is to reduce the dimensionality of each tag dataset while maintaining the  variance amongst the datasets. Then, the group of songs belonging to each tag can be separable. 

We utilized PCA to maintain a variance of 0.9 across our tag datasets, which provided us with enough leeway to reduce our eight features to six principal components, reducing our dataset sizes from (1000,8) to (1000,6). Method 1: Multilayer Perceptron (MLP)

In our first method, we assume that official Spotify playlists (curated by Spotify) that represent our social tags (ie. “Chill Mix”, “Party Mix”, etc) provide accurate representations of the feature values for each tag. For example, if we input a flattened feature vector of four songs from the “Chill Mix” playlist batch into our model, the ideal output vector that the model presents should have feature values equal to the average feature vector of the remaining songs in the “Chill Mix” playlist.  

In order to create a neural network that takes in a number of input song vectors and gives us an “ideal” average feature vector of the remaining songs, we need to create (x,y) data for each of the tag datasets, where:

  • x is a flattened vector of the input songs
  • y is the ground truth value, the average feature vector representing the rest of the songs in the playlist

In order to do this, we will batch our data with random reshuffling.

Generating Training Data through Batching w/ Random Reshuffling:

  1. Using a playlist of S song vectors, we create S//10 batches of size 10. So if we had a playlist of 50 song vectors, we would have 5 batches, each containing 10 vectors.

  1. Within each batch, create the (x,y) pairs: we will randomly select num_input_songs = 4 songs without replacement and calculate an average feature vector of the remaining songs (num_output_songs = 6).

  1. The model repeats step 2, num_sub_batches number of times. Sub_batches is the number of times (x,y) pairs are sampled within the batch.

Note: After training our model on various hyperparameter values, we found that num_input_songs = 4, num_output_songs = 6, num_sub_batches = 5 performed the best.

The reason we use this method of random reshuffling is so that we can see the effects of each set of input vectors on the output vector of average features. These (x,y) pairs are the training data that will be passed into our neural network.

Figure 4: Displays the random reshuffling batching method.

Training MLP:

  1. We trained the model by passing in the (x,y) data pairs obtained above into the neural network.
  2. The input data (which is a flattened feature vector of the dataset songs) goes through two hidden layers: the first constant activation function layer, a nonlinear activation transformation (ReLU), and a second constant activation function layer to obtain the predicted output feature vector.
  3. MSE loss is calculated between the predicted output feature vector (which is received from our neural network) and the actual vector (y, the average feature vector calculated during batching). Then, we used adaptive gradient stochastic to improve the model parameters after each epoch.

Figure 5. Displays the multilayer perceptron architecture.

Method 2: Recurrent Neural Network (RNN)

To motivate our second method, we make the assumption that listening to music and generating a “steerable” playlist is a sequential process.

Previous works that attempted to leverage song order for generating song recommendations found RNN’s to perform the best, which is why we decided to utilize RNN to create our user-steerable playlist in our second method.

Our goal is to create a model that takes a number of input songs from an incomplete playlist and gives us candidate output songs that complete the playlist while retaining the correct social tag. The motivation behind utilizing a RNN model is that users often input songs into a playlist sequentially and based on the previously existing songs. We need to first create (x,y) data for each of the tag datasets, where:

  • x is a matrix of the input songs
  • y is the matrix of output songs representing the remaining songs in the playlist in sequence

In order to do this, we will batch our data without random reshuffling. We do not randomly reshuffle because we want to maintain order in our data (for our RNN to find sequences).

Generating Training Data through Batching without Random Reshuffling:

  1. Using a playlist of S song vectors, we create S//10 batches of size 10. So if we had a playlist of 50 song vectors, we would have 5 batches, each containing 10 vectors.

  1. Within each batch, create the (x,y) pairs: we will sequentially select num_input_songs = 4 songs without replacement and create a matrix of remaining songs in the batch (num_output_songs = 6).

Note: After training our model on various hyperparameter values, we found that num_input_songs = 4, num_output_songs = 6 performed the best.

Training RNN:

Previous architectures found that RNN’s worked successfully when trained only on sequential data and no other additional metadata. Furthermore, it has been found that the ordering of the input data did not seem to affect the performance of the RNN, so we arbitrarily ordered our input data based on the batching method above (Schedl, 2019).  

  1. We trained the model by passing in the (x,y) data pairs obtained above into the RNN.
  2. The input data (a matrix of 4 input songs from the dataset) goes through two hidden layers: an RNN layer and a fully-connected layer to obtain a predicted output feature matrix of 6 songs.
  3. In order for the RNN to create context to predict vectors for remaining songs (songs 5-10) from input song vectors 1-4, it compares the predicted and actual vector values for songs 1-4. For example, the RNN initially inputs the vector for song 1 and predicts the vector for song 2. Then, it can compare this to the actual vector value of song 2. This creates context with which the RNN can predict song 5 from song 4, song 6 from the predicted song 5, and so on. The details of how the RNN predicts the remaining 6 output song vectors is detailed in Figure 6 (below).
  4. MSE loss is calculated between the predicted output song matrix (which is received from our recurrent neural network) and the actual song matrix y (the sequence of songs obtained from batching). Then, we used adaptive gradient stochastic to improve the model parameters after each epoch.

Testing our Models and Comparing the Results:

We took one chill, one sad, one happy, and one party playlist that have never been seen before and tested our models on them. For each playlist:

  1. We batched the playlist into batches of size 10, the first 4 songs in each batch being the inputs and the last 6 being the desired outputs (y-actual)
  2. Next we,
  1. input the batches into the MLP model and obtain an average output vector (y-predicted)
  2. input the batches into the RNN model and obtain an output matrix of six songs (y-predicted) where each row represents one song
  1. Finally, we compute RMSE between each row in y-actual (each row being the features of one output song) and
  1. The MLP output average feature vector
  2. Each row in the RNN output matrix

Our results showed that both MLP and RNN performed relatively the same. However, intuitively, song listening is a sequential process (i.e. listening to songs in an album). So, with the aim of our project being to create a steerable playlist, it makes sense to use a recurrent neural network to generate our output songs.

Figure 12: Scatter plot of the RMSE values for each batch for each tag playlist. Both MLP and RNN had similar RMSE values.

Obtaining Final Results:

Thus far, the outputs received by the

  • Multilayer Perceptron: one normalized feature vector representing the output song based on the user’s input songs and social tag
  • Recurrent Neural Network: a normalized feature matrix representing the next six output songs in sequence, based on the user’s input songs and social tag

Next, for

  • Multilayer Perceptron: we find six songs from the original dataset (ie. a combination of chill, party, happy, and sad songs) that minimize the Euclidean distance between the neural network’s predicted vector.
  • Recurrent Neural Network: we find six songs from the original dataset that minimize the Euclidean distance from each of the six songs represented in the network’s predicted output feature matrix

Note: The original dataset upon which we make these comparisons is denormalized and PCA fit, because that is the format of our output data.

Summary:

While song recommendation systems are a common machine learning project, our project is unique in its aim to generate a user-steerable playlist that is based on input songs and a social tag that the user provides.

We explored two different neural network architectures: Multilayer Perceptron and Recurrent Neural Network. Each of these neural networks was chosen based on different assumptions: our MLP model assumes that the ideal completion of a playlist is the average vector of the output (completed) songs, while the RNN model accounts for the fact that users typically complete/add songs to playlists sequentially based on existing songs in the playlist.

After training and validating both our models, we concluded that neither model is superior as both models performed approximately equally well and had similar RMSE values. However, we choose a RNN model because song playlist completion is a sequential process and this sequential aspect should be considered.
